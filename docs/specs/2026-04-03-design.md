## 一、项目概述

### 1.1 项目背景

本项目为课程第三阶段任务，在第二阶段消费过程描述的基础上，实现一个可运行的一卡通模拟消费终端系统。系统以"文件模拟卡片"为核心设计理念，通过打开/关闭文件来模拟物理卡片的放置与拔出，还原学生在食堂、超市等场所刷卡消费的完整业务流程。

### 1.2 项目目标

- 实现一个带图形界面的消费终端程序，包含显示区与完整按键区
- 以 JSON 文本文件作为卡片载体，实现卡片数据的读写管理
- 实现扣款前的辅助运算功能（加法、减法，可选乘法）
- 实现一个独立的充值发卡站程序，支持发卡与圈存充值
- 设计并书面描述充值中心与消费端的金额结算方案

### 1.3 技术选型

| 项目 | 选型 | 说明 |
|------|------|------|
| 编程语言 | Python 3.x | 简洁易用，库生态丰富 |
| GUI 框架 | tkinter（标准库） | 无需额外安装，跨平台，满足终端界面需求 |
| 卡片格式 | JSON 文本文件 | 可读性强，便于手动修改模拟换卡 |
| 交易记录 | JSON 文本文件 | 本地存储，记录每笔消费/充值流水 |
| 结算记录 | JSON 文本文件 | 各商户账单汇总，用于结算方案演示 |

## 二、系统整体架构

### 2.1 系统由两个独立程序组成

整个系统由两个独立的 Python 程序构成，分别模拟现实中的两类设备：

| 程序 | 文件名 | 对应现实设备 | 主要功能 |
|------|--------|-------------|---------|
| 消费终端 | `terminal.py` | 食堂/超市 POS 刷卡机 | 读卡、运算、扣款、显示余额 |
| 充值发卡站 | `issuer.py` | 校内自助充值机/一卡通中心 | 新卡发行、账户充值（圈存） |

两个程序共享相同的卡片文件格式和交易记录目录，通过文件系统进行数据交换，无需网络通信，架构简洁、部署方便。

### 2.2 文件结构

```
project/
├── terminal.py          # 消费终端主程序
├── issuer.py            # 充值发卡站主程序
├── card_manager.py      # 卡片文件读写模块（两个程序共用）
├── calculator.py        # 辅助运算模块
├── logger.py            # 交易记录模块
├── data/
│   ├── cards/           # 存放卡片文件（每张卡一个 .json 文件）
│   │   ├── 2024001.json
│   │   └── 2024002.json
│   └── logs/            # 交易流水记录
│       ├── terminal_log.json   # 消费终端交易记录
│       └── issuer_log.json     # 充值发卡记录
└── README.txt           # 运行说明
```

## 三、数据设计

### 3.1 卡片文件格式（.json）

每张卡对应一个 JSON 文件，文件名即卡号（如 `2024001.json`）。文件内容示例如下：

```json
{
  "card_id":    "2024001",
  "name":       "张三",
  "balance":    85.50,
  "status":     "active",
  "created_at": "2026-04-01"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `card_id` | 字符串 | 卡号，与文件名一致 |
| `name` | 字符串 | 持卡人姓名 |
| `balance` | 浮点数 | 当前余额（元），扣款/充值后更新 |
| `status` | 字符串 | 卡片状态：`active`（正常）/ `suspended`（挂失） |
| `created_at` | 字符串 | 建卡日期 |

### 3.2 交易记录格式

每笔交易追加写入对应的 log 文件，便于后续查询与结算统计：

```json
{
  "tx_id":         "TX20260401120500",
  "card_id":       "2024001",
  "type":          "consume",
  "amount":        15.00,
  "balance_after": 70.50,
  "merchant":      "一食堂",
  "time":          "2026-04-01 12:05:00"
}
```

| 字段 | 说明 |
|------|------|
| `tx_id` | 交易流水号，格式：`TX` + 年月日时分秒 |
| `type` | 交易类型：`consume`（消费）/ `recharge`（圈存充值） |
| `amount` | 本次交易金额（元） |
| `balance_after` | 交易完成后卡内余额 |
| `merchant` | 消费商户名称（如：一食堂、校内超市） |

## 四、模块设计

### 4.1 `card_manager.py` — 卡片管理模块

负责卡片文件的所有读写操作，是两个程序的核心公共模块。

```python
class CardManager:
    def load_card(filepath) -> dict
        # 读取卡片 JSON 文件，返回卡片数据字典
        # 对应"放卡"操作（O 键 / 文件选择）

    def save_card(filepath, card_data) -> None
        # 将更新后的卡片数据写回 JSON 文件
        # 扣款或充值成功后调用

    def deduct(card_data, amount) -> (bool, str)
        # 执行扣款：校验余额是否充足
        # 返回 (成功标志, 提示信息)

    def recharge(card_data, amount) -> (bool, str)
        # 执行圈存充值，更新 balance 字段

    def create_card(card_id, name, initial_balance) -> dict
        # 创建新卡数据结构，用于发卡
```

```python

        self.current_num = ''  # 当前正在输入的数字

    def backspace() -> str
        # 回退一位，删除最后一个输入字符

    def calculate() -> float
        # 计算 expression 中存储的表达式，返回最终金额
        # 此金额将作为扣款金额传入 CardManager.deduct()
```

## 六、核心业务流程

### 6.1 消费扣款流程

以下为完整的一次消费过程，对应真实刷卡消费的各个环节：

| 步骤 | 操作 | 程序响应 |
|------|------|---------|
| 1 | 点击「放卡（O）」按钮，选择卡片文件 | 读取 JSON 文件，显示持卡人姓名和当前余额，状态栏变为「已放卡」 |
| 2 | 通过数字键和运算键输入消费金额 | 显示区实时更新，支持多件商品累加（如：5.5 + 8.0 + 12.0） |
| 3 | 点击「扣款」按钮 | Calculator 计算总金额，CardManager 校验余额是否充足 |
| 4a（余额充足） | — | 余额扣减，写回 JSON 文件；Logger 记录交易；显示「扣款成功，余额 XX 元」 |
| 4b（余额不足） | — | 不修改文件，显示「余额不足，当前余额 XX 元，请先充值」 |
| 5 | 点击「拔卡（C）」按钮 | 关闭卡片文件（内存清空），状态栏变为「未放卡」，显示区清零 |

### 6.2 发卡与圈存流程

| 步骤 | 操作 | 程序响应 |
|------|------|---------|
| 1（发卡） | 输入卡号、姓名、初始金额，点击「生成卡片」 | 在 `data/cards/` 创建对应 JSON 文件，写入初始数据 |
| 2（圈存） | 选择卡片文件，输入充值金额，点击「圈存充值」 | 读取文件，`balance` 字段增加充值金额，写回文件，记录充值日志 |

## 七、充值与结算方案设计

### 7.1 方案概述

本方案描述校园一卡通系统中，充值中心的资金如何与各消费端（食堂、超市等商户）进行结算。整个资金流转分为三个环节：充值上卡、消费扣款、商户结算。

### 7.2 充值上卡（圈存）

学生通过支付宝、微信或银行卡向校园一卡通账户充值后，资金进入学校统一管理的充值中心账户。充值的钱并不立即写入物理卡片，学生需将卡片放置到自助充值机（即本系统的充值发卡站）上执行圈存操作，将资金从充值中心转移到卡片余额中。

- **圈存前**：资金存放在充值中心账户，卡片余额不变
- **圈存后**：充值中心扣除对应金额，卡片 `balance` 字段增加，即"钱从云端下载到卡上"

### 7.3 消费扣款

学生在食堂、超市等消费终端刷卡时，系统从卡片文件中直接扣减 `balance` 字段，并在本地交易日志中记录本笔消费的金额、时间和商户信息。消费时资金暂不实时划转给商户，由各终端本地记录待结算金额。

### 7.4 商户结算

商户结算按周期进行（例如每日或每周），流程如下：

1. 各消费终端汇总本结算周期内的交易日志，统计该商户应收总金额
2. 将汇总数据提交至充值中心（在模拟系统中，即读取 `terminal_log.json` 中该商户的记录）
3. 充值中心核对数据后，从学生消费总额中划拨对应金额至商户账户
4. 结算完成，各终端交易记录归档，开始下一结算周期

### 7.5 资金流转示意

```
学生网上充值
      │  ① 支付宝/微信充值
      ▼
充值中心账户（学校）
      │  ② 学生持卡到充值机圈存
      ▼
  卡片余额（balance）
      │  ③ 刷卡消费，终端记录流水
      ▼
终端交易日志（本地）
      │  ④ 周期结算，汇总上报
      ▼
充值中心划款至商户账户
```

| 环节 | 资金去向 | 数据变化 |
|------|---------|---------|
| 充值 | 个人 → 充值中心 | 充值中心账户余额增加 |
| 圈存 | 充值中心 → 卡片 | 充值中心减少，`card.balance` 增加 |
| 消费 | 卡片 → 待结算 | `card.balance` 减少，终端日志新增记录 |
| 结算 | 充值中心 → 商户 | 充值中心划款，商户到账 |

### 8.2 测试场景

- **正常消费**：放卡 → 输入多件商品金额相加 → 扣款 → 验证文件余额变化 → 拔卡
- **余额不足**：输入超过余额的金额 → 验证不扣款且提示正确
- **发卡 + 圈存**：新建卡片文件 → 充值 → 验证 `balance` 字段更新
- **交易记录验证**：查看 log 文件内容，确认每笔交易均有完整记录
- **手动换卡**：修改 JSON 文件中的卡号和金额，模拟换一张卡重新消费
