## 一、项目概述

### 1.1 项目背景

本项目为课程第三阶段任务，在第二阶段消费过程描述的基础上，实现一个可运行的一卡通模拟消费终端系统。系统以"文件模拟卡片"为核心设计理念，通过打开/关闭文件来模拟物理卡片的放置与拔出，还原学生在食堂、超市等场所刷卡消费的完整业务流程。

### 1.2 项目目标

- 实现一个带图形界面的消费终端程序，包含显示区与完整按键区
- 以 JSON 文本文件作为卡片载体，实现卡片数据的读写管理
- 实现扣款前的辅助运算功能（加法、减法，可选乘法）
- 实现一个独立的充值发卡站程序，支持发卡与圈存充值
- 设计并书面描述充值中心与消费端的金额结算方案

### 1.3 技术选型

| 项目 | 选型 | 说明 |
|------|------|------|
| 编程语言 | Python 3.x | 简洁易用，库生态丰富 |
| GUI 框架 | tkinter（标准库） | 无需额外安装，跨平台，满足终端界面需求 |
| 包管理 | uv（推荐）/ pip | uv 为推荐的运行工具，也可使用传统 `python` 命令 |
| 卡片载体 | JSON 文本文件 | 可读性强，便于手动修改模拟换卡 |
| 持久化存储 | SQLite | 轻量级数据库，保证卡片数据和交易记录的一致性 |
| 单元测试 | unittest（标准库） | 对核心模块编写自动化测试 |

整个系统由两个独立的 Python 程序构成，分别模拟现实中的两类设备：

## 三、数据设计

### 3.1 SQLite 数据库设计

使用 SQLite 作为持久化存储，保证卡片数据和交易记录的数据一致性。

**cards 表**：

```sql
CREATE TABLE cards (
    card_id    TEXT PRIMARY KEY,
    name       TEXT NOT NULL,
    balance    REAL NOT NULL DEFAULT 0,
    status     TEXT NOT NULL DEFAULT 'active',
    created_at TEXT NOT NULL
);
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `card_id` | TEXT | 卡号，主键 |
| `name` | TEXT | 持卡人姓名 |
| `balance` | REAL | 当前余额（元），扣款/充值后更新 |
| `status` | TEXT | 卡片状态：`active`（正常）/ `suspended`（挂失） |
| `created_at` | TEXT | 建卡日期 |

**transactions 表**：

```sql
CREATE TABLE transactions (
    tx_id         TEXT PRIMARY KEY,
    card_id       TEXT NOT NULL REFERENCES cards(card_id),
    type          TEXT NOT NULL,
    amount        REAL NOT NULL,
    balance_after REAL NOT NULL,
    merchant      TEXT DEFAULT '',
    time          TEXT NOT NULL
);
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `tx_id` | TEXT | 交易流水号，主键，格式：`TX` + 年月日时分秒 |
| `card_id` | TEXT | 关联卡号 |
| `type` | TEXT | 交易类型：`consume`（消费）/ `recharge`（圈存充值） |
| `amount` | REAL | 本次交易金额（元） |
| `balance_after` | REAL | 交易完成后卡内余额 |
| `merchant` | TEXT | 消费商户名称（如：一食堂、校内超市） |
| `time` | TEXT | 交易时间 |

### 3.2 卡片文件格式

每张卡导出为一个文件，**文件名即卡号，无后缀**（如 `2024001`），内容为 JSON 格式。用于卡片在不同程序间的传递和手动修改模拟。文件内容示例如下：

```json
{
  "card_id":    "2024001",
  "name":       "张三",
  "balance":    85.50,
  "status":     "active",
  "created_at": "2026-04-01"
}
```

JSON 文件是卡片的导出/导入格式，真实数据以 SQLite 数据库为准。

### 3.3 数据一致性策略

**核心原则：SQLite 是唯一权威数据源。**

当卡片文件（JSON）与数据库余额不一致时（例如卡在另一个终端消费过，或充值后 JSON 文件未同步），遵循以下规则：

- **余额以 SQLite 为准**：拖入卡片时，仅从 JSON 中读取 `card_id`，去 SQLite 查询最新余额进行校验和扣款
- **每次交易后自动同步**：扣款或充值成功后，立即导出最新数据覆盖 JSON 文件，保证下次刷卡时卡面数据是最新的
- **JSON 的 `balance` 字段仅作参考**：不参与任何金额校验逻辑，仅用于界面上显示"卡面余额"与"实际余额"的对比（可选）

这与真实一卡通逻辑一致：POS 机联网查询后台余额，不信任卡片上存储的余额。

### 3.4 交易记录查询

交易记录存储在 SQLite 的 `transactions` 表中，支持以下查询：

- 按卡号查询交易历史
- 按商户统计累计消费金额（用于结算方案演示）
- 按交易类型过滤（消费/充值）

## 六、核心业务流程

### 6.1 消费扣款流程

以下为完整的一次消费过程，对应真实 POS 机的操作顺序：先输入金额，再刷卡，自动完成扣款。

| 步骤 | 操作 | 程序响应 |
|------|------|---------|
| 1 | 通过数字键和运算键输入消费金额 | 显示区实时更新，支持多件商品累加（如：5.5 + 8.0 + 12.0） |
| 2 | 点击「确定」按钮 | Calculator 计算总金额，显示区提示「金额 XX 元，请刷卡」，进入等待刷卡状态 |
| 3 | 拖拽卡片 JSON 文件到终端窗口（松开鼠标） | 自动导入 JSON 到 SQLite，校验卡片状态和余额 |
| 4a（余额充足） | — | 余额扣减，写回 SQLite；Logger 记录交易；导出更新后的 JSON 文件；结果显示「扣款成功，持卡人 XX，扣款 XX 元，余额 XX 元」 |
| 4b（余额不足） | — | 不修改数据，显示「余额不足，持卡人 XX，当前余额 XX 元，请先充值」 |
| 5 | 输入数字或其他操作 | 自动清除上次结果，回到步骤 1，开始下一笔交易 |

### 6.2 发卡与圈存流程

| 步骤 | 操作 | 程序响应 |
|------|------|---------|
| 1（发卡） | 输入卡号、姓名、初始金额，点击「生成卡片」 | 写入 SQLite 数据库，并导出 JSON 文件到 `data/cards/` |
| 2（圈存） | 输入充值金额 → 拖拽卡片 JSON 文件到窗口 | 自动导入 JSON → SQLite 更新 `balance` → 导出更新后的 JSON 文件 → 记录充值日志 → 显示充值结果 |

## 七、充值与结算方案设计

### 7.1 方案概述

本方案描述校园一卡通系统中，充值中心的资金如何与各消费端（食堂、超市等商户）进行结算。整个资金流转分为三个环节：充值上卡、消费扣款、商户结算。

### 7.2 充值上卡（圈存）

学生通过支付宝、微信或银行卡向校园一卡通账户充值后，资金进入学校统一管理的充值中心账户。充值的钱并不立即写入物理卡片，学生需将卡片放置到自助充值机（即本系统的充值发卡站）上执行圈存操作，将资金从充值中心转移到卡片余额中。

- **圈存前**：资金存放在充值中心账户，卡片余额不变
- **圈存后**：充值中心扣除对应金额，卡片 `balance` 字段增加，即"钱从云端下载到卡上"

### 7.3 消费扣款

学生在食堂、超市等消费终端刷卡时，系统从卡片文件中直接扣减 `balance` 字段，并在本地交易日志中记录本笔消费的金额、时间和商户信息。消费时资金暂不实时划转给商户，由各终端本地记录待结算金额。

### 7.4 商户结算

商户结算按周期进行（例如每日或每周），流程如下：

1. 各消费终端汇总本结算周期内的交易日志，统计该商户应收总金额
2. 将汇总数据提交至充值中心（在模拟系统中，即从 SQLite 的 `transactions` 表查询该商户的记录）
3. 充值中心核对数据后，从学生消费总额中划拨对应金额至商户账户
4. 结算完成，各终端交易记录归档，开始下一结算周期

### 7.5 资金流转示意

```
学生网上充值
      │  ① 支付宝/微信充值
      ▼
充值中心账户（学校）
      │  ② 学生持卡到充值机圈存
      ▼
  卡片余额（balance）
      │  ③ 刷卡消费，终端记录流水
      ▼
终端交易日志（本地）
      │  ④ 周期结算，汇总上报
      ▼
充值中心划款至商户账户
```

| 环节 | 资金去向 | 数据变化 |
|------|---------|---------|
| 充值 | 个人 → 充值中心 | 充值中心账户余额增加 |
| 圈存 | 充值中心 → 卡片 | 充值中心减少，`card.balance` 增加 |
| 消费 | 卡片 → 待结算 | `card.balance` 减少，终端日志新增记录 |
| 结算 | 充值中心 → 商户 | 充值中心划款，商户到账 |
